# Design Parser Reference

This document explains how `designparser.de` can be used as an external design
rule reference for SvS UI Nova.

Reference:

- [designparser.de](https://designparser.de/)

## Purpose

Design Parser is useful as a fast visual-rule cheatsheet for topics such as:

- spacing rhythm
- hierarchy and readability
- component density
- card and panel structure
- form layout
- optical balance

In this repo, it should be treated as a calibration source, not as the design
system itself.

## Precedence

When working in this repository, follow this order:

1. Existing components under `src/SvsUiNova/components`
2. Existing semantic tokens and theme variables in `src/styles/theme.css`
3. Existing detailed token definitions in `src/SvsUiNova/styles.css`
4. Existing guidelines under `guidelines/` and `ai-context/`
5. External references such as Design Parser

If Design Parser suggests something that conflicts with the repo's component
API, token model, or semantic theming, prefer the repo.

## How To Use Design Parser Here

Use Design Parser to:

- pressure-test spacing choices when repo guidance is still thin
- improve visual hierarchy and grouping
- sanity-check card, dialog, table, and form density
- improve optical alignment when literal math looks off
- refine layouts generated by AI tools

Do not use Design Parser to:

- introduce raw values where semantic tokens already exist
- create a parallel naming system for colors, spacing, or typography
- replace SvS product theming with a generic palette model
- bypass existing component imports and composition rules

## Repo Mapping

Translate external design rules into repo-native implementation:

- spacing advice -> `gap-*`, `space-y-*`, or repo token variables
- color advice -> semantic roles such as `background`, `card`, `primary`,
  `muted`, `border`, and status colors
- hierarchy advice -> existing heading structure and `svs-*` typography classes
- component advice -> `Card`, `Button`, `Input`, `Dialog`, `Table`, `Tabs`,
  `Popover`, and related repo components
- layout advice -> existing page, section, and surface patterns in demos and
  component docs

## AI Guidance

When an AI system uses Design Parser ideas in this repo, it should:

1. convert generic advice into SvS UI Nova tokens and components
2. preserve semantic color usage and product theme compatibility
3. preserve current import paths and public component APIs
4. prefer existing utilities and classes over custom CSS
5. label any gap-filling recommendation as a recommendation, not as an
   established repo rule

## Good Example

"Use a clearer visual hierarchy for this settings panel" is a good use of
Design Parser when the implementation becomes:

- a `Card` surface
- `p-4` or `p-6` padding from the repo scale
- `space-y-4` or `space-y-6` between groups
- semantic text colors
- existing `Button`, `Switch`, and `Input` components

## Bad Example

"Design Parser uses 18px here, so add `padding: 18px` and `border-radius:
13px` inline" is not acceptable when the repo already has a spacing and radius
system.

## Recommendation

Use Design Parser as a review companion and refinement aid. Keep SvS UI Nova as
the canonical source of truth.
