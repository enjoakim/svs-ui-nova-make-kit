# Intention First

Design that starts with experience, not components. In SVS Design, we begin with understanding what users should feel, understand, and do—then use the design system to get there with quality and consistency.

---

## Introduction

**In SVS Design, we do not start with components, tokens, or rules. We start with intention.**

Every view, flow, and moment in the product has a purpose—and a feeling. That is what we design for first.

**Core Philosophy:**
> The design system is a tool to realize intention, not a replacement for creative thinking. We design experiences first, then translate them into system components.

---

## What Do We Mean by Intention?

Intention is about what we want the user to **feel**, **understand**, and **do**.

### Feel
What emotional state should the user experience?

**Examples:**
- Safe and secure
- Engaged and excited
- In control and confident
- Calm and focused
- Delighted and surprised
- Supported and cared for

**Design Impact:**
Emotional intent drives choices in color, motion, spacing, and tone. A moment meant to feel calm uses different design tools than one meant to feel exciting.

---

### Understand
What should be clear to the user at this moment?

**Examples:**
- What matters most right now
- What happens next in the flow
- What is expected of them
- What their options are
- Where they are in the journey
- Why this matters to them

**Design Impact:**
Clarity of understanding drives hierarchy, information architecture, labeling, and content structure. Users can't act confidently if they don't understand.

---

### Do
What action should the user take?

**Examples:**
- Act (place bet, submit form, confirm choice)
- Pause (review information, consider options)
- Explore (discover features, browse games)
- Confirm (verify decision, acknowledge info)
- Navigate (move forward, go back, jump to section)
- Wait (loading, processing)

**Design Impact:**
The desired action influences CTA prominence, interaction patterns, navigation structure, and feedback mechanisms.

---

**The Foundation:**
> Intention sets the direction. The design system helps us get there—consistently and with quality.

---

## From Intention to Design Decisions

When the intention is clear, better design decisions follow.

### Examples in Practice

#### "This moment should feel calm and focused"
**Intention:** User needs to concentrate on an important decision

**Design Decisions:**
- → Restrained use of color
- → Low visual noise
- → Clear hierarchy
- → Ample white space
- → Minimal distractions
- → Gentle, purposeful motion

**Components Used:**
- Simple button hierarchy (one primary action)
- Clean card layouts
- Neutral surface colors
- Quiet typography scale

---

#### "This is a peak moment"
**Intention:** Celebrate user success or highlight critical decision

**Design Decisions:**
- → Stronger contrast
- → Accent color prominence
- → Clear call to action
- → Engaging motion
- → Focused attention
- → Emotional resonance

**Components Used:**
- Bold primary buttons
- Accent color backgrounds
- Celebratory animations
- Large, confident typography

---

#### "This is informational, not emotional"
**Intention:** Provide facts without adding pressure or excitement

**Design Decisions:**
- → Neutral color roles
- → Low visual weight
- → Clear structure
- → Readable typography
- → No emotional triggers
- → Calm, stable presentation

**Components Used:**
- Text-based layouts
- Neutral surfaces
- Simple data displays
- Minimal interactive elements

---

### Design System as Tools

**Key Principle:**
> Color, typography, spacing, and components are used as tools to support the intention—not as the starting point.

**Don't Start With:**
- "Which component should I use?"
- "What's the correct token?"
- "What does the system allow?"

**Start With:**
- "What should the user feel here?"
- "What needs to be understood?"
- "What action do we want to enable?"

**Then Ask:**
- "Which components best serve this intention?"
- "How can the system help realize this experience?"
- "What customization supports the goal?"

---

## Right Tool, Right Phase

To enable strong experiences, the design system needs to be used at the right time.

### Phase 1: Exploration / Concept

**Focus:**
- Flow, tempo, and feeling
- User journey and emotional arc
- Core interactions and moments
- Problem-solution fit

**How to Work:**
- **Intention is expressed in words** - Write down what users should feel, understand, do
- **System rules are secondary** - Don't constrain creativity with components yet
- **Focus on the experience** - Sketch, prototype, explore freely
- **Test the concept** - Validate intention with users

**Tools:**
- Wireframes and sketches
- User journey maps
- Storyboards
- Rapid prototypes
- Written intention statements

**System Role:**
The design system is a reference, not a constraint. Explore without limits.

---

### Phase 2: Chosen Direction

**Focus:**
- Refine and validate the concept
- Align with stakeholders
- Shape the solution
- Prepare for detailed design

**How to Work:**
- **The intention is confirmed** - Agreement on what users should feel/understand/do
- **The concept is shaped and aligned** - Clear direction established
- **Begin system alignment** - Start considering how system supports intention
- **Identify needed components** - Map intention to available patterns

**Tools:**
- Higher-fidelity prototypes
- User testing
- Stakeholder reviews
- Component mapping

**System Role:**
The design system becomes a guide. Start seeing how it can help.

---

### Phase 3: Translation into SVS Design

**Focus:**
- Quality and consistency
- Scalability and maintainability
- Technical feasibility
- Production readiness

**How to Work:**
- **The system is used to ensure quality, consistency, and scalability**
- **Deviations are allowed** - Intentional and well-motivated
- **Document decisions** - Especially when deviating from system
- **Maintain intention** - Don't lose the original vision in translation

**Tools:**
- Design system components
- Design tokens
- Pattern library
- Component documentation
- Production designs

**System Role:**
The design system is the primary tool. Use it to implement with quality.

---

**Key Insight:**
> The design system should support creativity, not replace it.

## When to Deviate from the System

Deviations from the design system are allowed when:

### Intentionally Motivated
- The standard component doesn't serve the intention
- The user experience requires something different
- The context demands unique treatment
- Innovation requires exploration

### Well-Documented
- Reason for deviation is clear
- Alternative considered and rejected
- Impact on consistency is understood
- Team is aligned on the decision

### Examples of Valid Deviations

**Game-Specific Experiences:**
- Unique interaction patterns for specific games
- Special moments that need distinctive treatment
- Category-specific visual language

**Experimental Features:**
- Testing new concepts
- Innovation in user experience
- Learning what works for users

**Brand Moments:**
- Campaign-specific expressions
- Special events or celebrations
- High-impact marketing moments

## A Shared Responsibility

Working intention-first is a shared responsibility across design, tech, and product.

### This Means That:

#### We talk more about "why" than "which component"
**In Reviews:**
- ❌ "Why didn't you use the standard button?"
- ✓ "What should the user feel at this moment?"
- ✓ "Does this component serve the intention?"

**In Planning:**
- ❌ "Let's use a modal for this."
- ✓ "We want users to focus and confirm. What pattern best supports that?"

---

#### We treat the system as an enabler, not a rulebook
**In Design:**
- The system provides solutions, not restrictions
- Components are tools to realize intention
- Deviations are discussed, not forbidden

**In Development:**
- Understanding the "why" behind component choices
- Implementing with intention in mind
- Suggesting system alternatives when they better serve the goal

---

#### We allow exploration without pointing fingers
**In Exploration:**
- Creative freedom to find the right solution
- No premature optimization for system compliance
- Focus on user experience first

**In Feedback:**
- Curiosity about intention before critique
- Collaborative problem-solving
- Support for creative risks

---

**Shared Goal:**
> The goal is always the same: to create player experiences that feel considered, trustworthy, and engaging.

## How to Apply Intention-First Thinking

### Step 1: Define the Intention

**Ask:**
- What should users **feel** at this moment?
- What do they need to **understand**?
- What should they **do**?

**Document:**
Write clear intention statements for each key moment or screen.

**Example:**
> "At checkout, users should feel confident and in control. They need to understand exactly what they're paying for and what happens next. The primary action is to complete the purchase with ease."

---

### Step 2: Explore Solutions

**Without System Constraints:**
- Sketch multiple approaches
- Consider different patterns
- Think beyond existing components
- Focus on serving the intention

**Test and Validate:**
- Does this solution match the intention?
- Do users feel what we want them to feel?
- Is it clear what they should understand and do?

---

### Step 3: Map to System

**Find System Patterns:**
- Which components support this intention?
- Can existing patterns be used?
- What customization is needed?

**Document Decisions:**
- Why these components were chosen
- How they serve the intention
- Any deviations and their rationale

---

### Step 4: Implement with Quality

**Use System Tools:**
- Components, tokens, patterns
- Ensure consistency where appropriate
- Maintain quality standards

**Preserve Intention:**
- Don't lose the vision in translation
- Test that the feeling is preserved
- Verify clarity and usability

---

## Intention Statement Template

Use this template to articulate intention:

```
Context: [What is this screen/flow/moment?]

User Goal: [What is the user trying to accomplish?]

Should Feel: [Emotional state - calm, excited, confident, etc.]

Should Understand:
- [Key information point 1]
- [Key information point 2]
- [Key information point 3]

Should Do: [Primary action or next step]

Success Looks Like: [How do we know this worked?]
```

**Example:**

```
Context: First-time user deposit flow

User Goal: Add money to their account safely

Should Feel: Safe, confident, in control

Should Understand:
- Their payment information is secure
- Exactly how much they're depositing
- They can set limits for responsible gaming
- What happens after they deposit

Should Do: Complete deposit with confidence

Success Looks Like: User deposits without hesitation and feels good about it
```

## Cross-Functional Collaboration

### Design's Role
- Define and articulate intention
- Explore solutions that serve intention
- Map intention to system components
- Maintain vision through implementation

### Product's Role
- Clarify user needs and business goals
- Validate that intention aligns with objectives
- Support creative exploration
- Balance innovation with consistency

### Tech's Role
- Understand the "why" behind design decisions
- Suggest technical solutions that serve intention
- Implement with both quality and intention preserved
- Identify when system patterns don't fit

## Common Pitfalls to Avoid

### Starting with Components
❌ "Let's use a modal for this."
✓ "Users need to focus on this critical decision. What pattern best supports that?"

### Over-Constraining Early Exploration
❌ "That doesn't match our components."
✓ "Interesting approach. How does it serve the intention? Can we achieve this with the system?"

### Losing Intention in Translation
❌ "We had to change it to fit the system."
✓ "We used these system components because they serve the intention while maintaining consistency."

### Deviating Without Reason
❌ "I liked this better visually."
✓ "We deviated here because [specific user need] required [specific solution]."

---

## Measuring Success

### Intention Clarity
- Can team members articulate what users should feel/understand/do?
- Are intention statements documented for key flows?
- Do stakeholders agree on the intention?

### Design Quality
- Does the final design achieve the stated intention?
- Do users feel what we intended them to feel?
- Is the purpose clear to users?

### System Alignment
- Are system components used where appropriate?
- Are deviations intentional and documented?
- Does the system support creative solutions?

### Team Health
- Are we talking about "why" before "how"?
- Is exploration welcomed and supported?
- Do we collaborate effectively across disciplines?

---

## Summary

**Intention-First Principles:**

1. **Start with Experience** - Feel, understand, do before components
2. **System as Tool** - Use components to serve intention, not vice versa
3. **Right Time, Right Tool** - Explore freely, then apply system rigor
4. **Shared Responsibility** - Everyone owns the user experience
5. **Thoughtful Deviation** - Break rules when it serves the user

**The Promise:**
> When we start with intention and use the system wisely, we create experiences that feel considered, trustworthy, and engaging—every time.
